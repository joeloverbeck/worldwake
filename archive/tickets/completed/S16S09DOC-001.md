## Problem

The `S16S09GOLVAL-002` work exposed two process gaps:

1. a ticket can still be written as "tests only" even when that is only a hypothesis and current code may contradict the intended invariant, and
2. a golden can still be scoped around a broad downstream behavior when the cleaner architectural contract is an earlier causal boundary such as goal selection.

Those gaps encourage brittle tickets and over-broad goldens, which pulls the suite away from `docs/FOUNDATIONS.md` and toward scenario scripting.

## Assumption Reassessment (2026-03-20)

1. `tickets/README.md` already contains most of the intended policy. It explicitly says to update the ticket first when code and ticket assumptions diverge, not to leave a ticket marked `Engine Changes: None` or "tests only" when production scope is wrong, and not to use delayed downstream effects as proxies when a lower-layer assertion surface exists.
2. `docs/golden-e2e-testing.md` already contains most of the intended boundary guidance. The assertion hierarchy, ordering rules, and decision-trace guidance already push authors toward semantic surfaces instead of broad downstream proxies.
3. The remaining gap in those two docs is precision and discoverability, not missing architecture. They do not currently say the "earliest causal boundary" rule in one explicit sentence, and they do not cross-link the concrete `S16S09GOLVAL-002` example that demonstrates the narrowing.
4. `AGENTS.md` still lacks the explicit debugging fallback discovered during `S16S09GOLVAL-002`: decision traces may show the selected outcome while still lacking enough provenance to explain the concrete world facts keeping that branch alive, so engineers sometimes need lower-layer focused state/query tests.
5. The active governing spec for this family is `specs/S16-s09-golden-validation.md`. No `specs/S15-s09*` path exists in the current repo, so this ticket should anchor to the real S16 spec and the completed `S16S09GOLVAL-002` example.
6. This remains a documentation/process ticket only. Current evidence points to a documentation clarity gap, not a new production-code contradiction.
7. This ticket is not ordering-centric, but it should clarify how to describe boundary choice when later execution or durable state is only a downstream consequence of an earlier AI-selection contract.
8. Mismatch corrected: the original ticket overstated what was missing from `tickets/README.md` and `docs/golden-e2e-testing.md`. The correct scope is targeted clarification plus an `AGENTS.md` addition, not broad policy invention.

## Architecture Check

1. Minimal clarifications in the canonical docs are better than creating new policy files or duplicating rules. The architecture problem here is discoverability of the right boundary, not absence of an abstraction.
2. Explicitly naming "earliest causal boundary" is beneficial because it compresses several existing rules into one stable authoring principle without changing the underlying architecture.
3. Adding the lower-layer fallback note to `AGENTS.md` is beneficial because it keeps debugging aligned with the real layering: decision traces for reasoning first, focused state/query tests when the missing fact lives below the trace surface.
4. No backwards-compatibility shims, alias paths, or duplicate guidance sources should be introduced.

## Verification Layers

1. Ticket-authoring contract makes provisional scope explicit and points authors to the narrowest correct causal boundary -> `tickets/README.md`
2. Golden-testing guidance names the earliest-causal-boundary rule explicitly and links it to the real completed example -> `docs/golden-e2e-testing.md`
3. AI debugging guidance states when decision traces are insufficient on their own and lower-layer focused tests are required -> `AGENTS.md`
4. Command verification confirms the cited example tests and baseline suites still exist -> named cargo commands plus `scripts/verify.sh`

## What to Change

### 1. Tighten `tickets/README.md`

Add one explicit clarification near the existing reassessment/ordering rules that:

1. `Engine Changes: None` is a provisional assumption until reassessment confirms no production contradiction,
2. tickets should be rewritten when new coverage exposes an underlying architectural bug, and
3. if a golden can prove the intended contract at an earlier causal boundary, the ticket should not widen scope to later downstream behavior unless that later behavior is itself the contract.

### 2. Tighten `docs/golden-e2e-testing.md`

Add a short clarification in the existing assertion guidance to state:

1. prefer the earliest semantic boundary that proves the invariant,
2. do not couple a golden to later action execution when candidate generation, ranking, suppression, or selection is the real contract,
3. use `archive/tickets/completed/S16S09GOLVAL-002.md` as an example of narrowing from downstream behavior to the correct changed-conditions selection boundary.

### 3. Tighten `AGENTS.md`

Update the decision-trace debugging section to say explicitly:

1. decision traces are the first stop for AI reasoning,
2. if traces show the selected outcome but not the concrete world facts keeping that branch alive, drop to the shared lower-layer state/query tests instead of adding ad-hoc instrumentation,
3. a follow-up traceability ticket may be warranted when the missing information is architecturally important rather than merely inconvenient.
