# S16S09TRACE-001: Decision Trace — Danger Pressure Provenance

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes - `worldwake-ai` pressure derivation helpers, decision-trace payloads, trace capture, and trace-focused tests
**Deps**: `archive/tickets/completed/S16S09GOLVAL-002.md`, `specs/S16-s09-golden-validation.md`

## Problem

The current decision trace explains which goal won, but not the concrete hostility facts that made danger pressure non-zero. In `S16S09GOLVAL-002`, that meant the trace could show "still selected `ReduceDanger`" without showing which entities still counted as hostiles or why. That leaves a causality gap: AI behavior is visible, but the concrete local state that fed that behavior is not.

## Assumption Reassessment (2026-03-20)

1. The current trace model in `crates/worldwake-ai/src/decision_trace.rs` records `PlanningPipelineTrace`, `CandidateTrace`, `RankedGoalSummary`, `SelectionTrace`, and `InterruptTrace`, but `RankedGoalSummary` only stores `goal`, `priority_class`, and `motive_score`. There is no structured field for why a danger-scored goal got that motive, and the same thin summary shape is reused for interrupt challengers.
2. `derive_danger_pressure()` in `crates/worldwake-ai/src/pressure.rs` derives danger from concrete hostility surfaces exposed by the belief/planning view, specifically `current_attackers_of()` and `visible_hostiles_for()`, plus the actor's wounded/incapacitated state and threshold bands. The current helper returns only the final `Permille`, so trace capture cannot reuse a structured assessment today.
3. The planner snapshot already stores the needed hostility surfaces in `crates/worldwake-ai/src/planning_snapshot.rs` (`hostile_targets`, `visible_hostiles`, `current_attackers`), and the runtime/snapshot query implementations already filter them through current liveness/locality rules in `crates/worldwake-sim/src/per_agent_belief_view.rs` and `crates/worldwake-ai/src/planning_state.rs`. The missing substrate is therefore not world data collection but a reusable typed danger assessment plus trace exposure of it.
4. Existing focused coverage proves pressure behavior and dead-hostile filtering, but not trace provenance:
   - `crates/worldwake-ai/src/pressure.rs`: `danger_pressure_is_zero_without_thresholds_or_threats`, `danger_pressure_uses_threshold_bands_monotonically`, `danger_pressure_promotes_single_attacker_when_wounded_or_incapacitated`, and `danger_pressure_promotes_visible_hostile_when_already_wounded`
   - `crates/worldwake-sim/src/per_agent_belief_view.rs`: `visible_hostiles_exclude_dead_believed_targets`
   - `crates/worldwake-ai/src/planning_state.rs`: `dead_hostiles_are_not_visible_or_actionable_in_snapshot_state`
   - `crates/worldwake-ai/src/agent_tick.rs`: `trace_planning_outcome_for_hungry_agent` and `trace_active_action_interrupt`
   - `crates/worldwake-ai/tests/golden_combat.rs`: `golden_defend_changed_conditions`
   None of these prove that decision traces expose the concrete hostility contributors that produced a danger-driven ranked or selected branch.
5. `AGENTS.md` currently tells developers to use decision traces first for AI debugging, but the current payload cannot answer "which hostiles still counted for danger on this tick?" or "why was danger medium/high/critical?" without reading code or reproducing the calculation by hand.
6. This ticket is about runtime `agent_tick` traceability, not planner behavior changes. The intended verification layers are focused pressure/trace-model tests, one focused runtime trace test on the existing `agent_tick` harness surface, and one real golden scenario assertion that exercises the populated payload in practice. Full action registries are not required for the focused runtime trace check because `agent_tick::tests::trace_planning_outcome_for_hungry_agent` already proves the narrow harness boundary.
7. This is not an ordering-centric ticket. The contract is provenance completeness for one decision tick, not relative action ordering between ticks.
8. No heuristic, filter, or planner behavior is being weakened. The missing architectural substrate is a reusable structured danger assessment; this ticket should add that substrate and expose it through the trace model rather than introducing trace-only reimplementations of danger logic.
9. The ticket should not widen the generic belief-view trait just for diagnostics. The cleaner design is to derive provenance from the same belief view/state already used by ranking and to attach that provenance to the ranked-goal trace surface that both planning selection and active-action interrupt traces already share.
10. Mismatch corrected: the gap is not "decision traces are missing"; it is "the shared ranked-goal/interrupt trace surface lacks concrete danger provenance even though the runtime already has enough local state to explain it."

## Architecture Check

1. The clean design is to add a reusable structured danger-assessment helper in `crates/worldwake-ai/src/pressure.rs`, then expose that data through the shared ranked-goal trace surface. That is cleaner than a planning-only debug blob because the same summary type is already used for both planning candidates and active-action interrupt challengers.
2. The provenance should be attached where the motive is reported, not as an unrelated side-channel string dump. A ranked goal that depends on danger should carry its own structured explanation. That keeps the trace model extensible if future tickets need similarly structured provenance for pain, social, or enterprise motives.
3. The provenance should be captured from the assembled planning/runtime state for that tick rather than by widening core belief-view interfaces with trace-only methods. That keeps the core abstraction small and avoids coupling runtime logic to diagnostics.
4. The payload should stay concrete and stable: the derived danger pressure, the classification inputs that chose the branch, and the entity ids grouped by hostility surface. That aligns with `docs/FOUNDATIONS.md`: concrete state over abstract scores, locality of information, and inspectable causality.
5. No backwards-compatibility shims, duplicate trace paths, or string-parsing-based diagnostics should be introduced.

## What to Change

### 1. Add a reusable structured danger assessment

Add a typed helper in `crates/worldwake-ai/src/pressure.rs` that computes the same danger result as `derive_danger_pressure()` but also preserves the concrete local inputs:

1. the derived danger pressure value used for the tick,
2. the hostility surfaces that fed it (`current_attackers`, `visible_hostiles`, and `hostile_targets` when available from the same view),
3. the actor-state predicates that affect escalation (`has_wounds`, `is_incapacitated`, and whether thresholds were available).

`derive_danger_pressure()` should delegate to this helper so there is still one canonical danger-calculation path.

### 2. Extend the trace model on the shared ranked-goal surface

Update `crates/worldwake-ai/src/decision_trace.rs` so `RankedGoalSummary` can carry structured motive provenance for danger-driven goals and interrupt challengers. Keep the field typed and optional; do not introduce debug strings or a planning-only side channel.

### 3. Populate the new payload from the real planning/runtime state

Update the decision-trace capture path in `crates/worldwake-ai/src/agent_tick.rs` so danger provenance is filled from the same read-phase belief view/state the planner actually used for that tick. Do not recompute against a looser or different view only for diagnostics.

### 4. Add focused trace tests

Add focused tests that prove:

1. the structured danger assessment preserves the hostility surfaces and derived pressure,
2. the ranked-goal trace payload is recorded when hostility exists,
3. dead or filtered hostiles do not appear as live contributors,
4. summary/dump helpers remain readable and stable enough for test diagnostics.

### 5. Add one real changed-conditions trace assertion

Strengthen `golden_defend_changed_conditions` so the post-death planning trace explicitly shows empty or non-contributing danger provenance rather than only proving the selected goal changed.
