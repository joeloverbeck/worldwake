# E10PROTRA-002: ResourceSource component in worldwake-core

**Status**: COMPLETED
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: Yes — new authoritative component registration
**Deps**: E10PROTRA-001 (CommodityKind already exists; no direct dep on 001 but logically follows)

## Problem

Harvesting must transfer goods from a concrete, depletable stock, not conjure them from a place tag. `ResourceSource` is the authoritative stored state that represents a finite supply of a commodity at a place or workstation, with optional regeneration metadata. Without this, the production system violates Principle 3 (concrete state) and the E10 conservation requirements.

## Assumption Reassessment (2026-03-10)

1. No `ResourceSource` type exists in the codebase — confirmed.
2. `CommodityKind` exists in `items.rs`. `Quantity` exists in `numerics.rs`. `Tick` exists in `ids.rs`. All are available for use.
3. `component_schema.rs` is the single authoritative declaration point for ECS components. It fans out into generated storage/query APIs in `component_tables.rs` and `world.rs`, while `delta.rs` keeps explicit `ComponentKind::ALL` and `ComponentValue` coverage tests that must be updated manually.
4. `ResourceSource` should be registerable on `EntityKind::Facility` and `EntityKind::Place`. The E10 spec says the source is attached to a place or workstation, so the kind predicate should accept both.
5. `Place` entities live in `Topology`, not in the component schema. That does not block attaching additional authoritative components to `EntityKind::Place` entities through `World`.
6. `NonZeroU32` is available from `std::num`.
7. `WorldTxn` already has a macro-driven simple-component replacement path. If `ResourceSource` is expected to be depleted and regenerated by systems, the transaction layer should support replacing it directly instead of forcing an ad hoc mutation path later.

## Architecture Check

1. `ResourceSource` is authoritative stored state per Principle 3. "Available workstations" and "can harvest" remain derived read-models.
2. Placing it in `worldwake-core` allows both `worldwake-systems` (harvest/regeneration logic) and `worldwake-sim` (affordance and validation queries) to depend on the same schema.
3. Regeneration timing should stay source-local. Storing only `regeneration_ticks_per_unit` would push E10PROTRA-007 toward a fragile global-tick-modulus rule or event-log backtracking. Adding `last_regeneration_tick: Option<Tick>` now keeps later regeneration explicit, deterministic, and local to the source.
4. The component belongs on `Facility | Place` entities. No alias entity kind or wrapper type should be introduced.
5. This ticket should also wire `ResourceSource` into the normal `WorldTxn` component replacement path now so later systems can mutate it through the same typed delta pipeline used elsewhere.

## What to Change

### 1. Add `ResourceSource` to `crates/worldwake-core/src/production.rs`

```rust
/// Concrete depletable stock of a commodity at a place or workstation.
#[derive(Clone, Debug, Eq, PartialEq, Serialize, Deserialize)]
pub struct ResourceSource {
    pub commodity: CommodityKind,
    pub available_quantity: Quantity,
    pub max_quantity: Quantity,
    pub regeneration_ticks_per_unit: Option<NonZeroU32>,
    pub last_regeneration_tick: Option<Tick>,
}
impl Component for ResourceSource {}
```

### 2. Register in `component_schema.rs`

Add `ResourceSource` to the `with_authoritative_components!` macro with kind predicate `|kind| kind == EntityKind::Facility || kind == EntityKind::Place`.

### 3. Schema fanout

Update imports and explicit coverage/tests in `component_tables.rs`, `world.rs`, and `delta.rs` as required by the schema-driven pattern.

### 4. Transaction-layer simple setter support

Add `ResourceSource` to `with_txn_simple_set_components!` so later production systems can replace the component through `WorldTxn::set_component_resource_source(...)` and emit typed `ComponentDelta` records without inventing a side path in E10PROTRA-007.

## Acceptance Criteria

### Tests That Must Pass

1. `ResourceSource` can be inserted/retrieved/removed on `EntityKind::Facility` entities through the `World` API.
2. `ResourceSource` can be inserted/retrieved/removed on `EntityKind::Place` entities through the `World` API.
3. `ResourceSource` insertion is rejected for `EntityKind::Agent` and other non-matching kinds.
4. `ResourceSource` round-trips through bincode.
5. `ResourceSource` with `regeneration_ticks_per_unit: None` and `Some(NonZeroU32)` both serialize correctly.
6. `ResourceSource` with `last_regeneration_tick: None` and `Some(Tick)` both serialize correctly.
7. `ComponentKind::ALL` and `ComponentValue` coverage include `ResourceSource`.
8. `WorldTxn::set_component_resource_source(...)` records a typed `ComponentDelta::Set` and updates the world on commit.
9. Existing suite: `cargo test -p worldwake-core`

### Invariants

1. `available_quantity` and `max_quantity` use `Quantity`; no raw integers and no floats.
2. Regeneration rate uses `Option<NonZeroU32>`; zero-tick regeneration is impossible.
3. `last_regeneration_tick` uses `Option<Tick>`; no raw `u64` timestamps inside the component.
4. `ResourceSource` is authoritative stored state, not a derived value.
5. Component kind predicate restricts to `Facility | Place`.
6. No `HashMap`/`HashSet` used.
