# E03ENTSTO-005: Deterministic Query API

**Status**: TODO
**Priority**: HIGH
**Effort**: Medium
**Engine Changes**: None
**Deps**: E03ENTSTO-004 (World struct with component CRUD)

## Problem

Simulation systems need to iterate over entities and their components in deterministic order. The spec requires:
- `entities()` in sorted id order
- `entities_with<T>()` for single-component queries
- `entities_with_component<T>()` returning `(EntityId, &T)` pairs
- Multi-component intersection queries with deterministic ordering
- Filtered queries that preserve base ordering

Since we use explicit typed tables (no generics on the query API), each query method is typed to a specific component.

## What to Change

### 1. Add query methods to `World`

In `crates/worldwake-core/src/world.rs`, add:

#### Basic entity iteration

```rust
/// All live entity ids in sorted order.
pub fn entities(&self) -> impl Iterator<Item = EntityId> + '_

/// All entity ids (live + archived) in sorted order.
pub fn all_entities(&self) -> impl Iterator<Item = EntityId> + '_

/// All live entities of a specific kind, sorted.
pub fn entities_of_kind(&self, kind: EntityKind) -> impl Iterator<Item = EntityId> + '_
```

#### Per-component queries (generated by macro for each table)

```rust
/// All entity ids that have a Name component (live entities only).
pub fn entities_with_name(&self) -> impl Iterator<Item = EntityId> + '_

/// All (EntityId, &Name) pairs for live entities, sorted by EntityId.
pub fn query_name(&self) -> impl Iterator<Item = (EntityId, &Name)> + '_
```

Same pattern for `agents`, `places`, and any other component table.

#### Multi-component intersection

```rust
/// Entities that have both Name and AgentData components (live only), sorted.
pub fn entities_with_name_and_agent(&self) -> impl Iterator<Item = EntityId> + '_

/// Query returning (EntityId, &Name, &AgentData) for entities with both.
pub fn query_name_and_agent(&self) -> impl Iterator<Item = (EntityId, &Name, &AgentData)> + '_
```

Multi-component intersection uses the smaller table as the driver and checks presence in the larger table(s). All results are in EntityId-sorted order because `BTreeMap` iteration is sorted.

#### Filtered queries

```rust
/// Filter any query with a predicate, preserving sorted order.
/// (This is just standard Iterator::filter — document that ordering is preserved.)
```

No special method needed — standard `Iterator::filter` on sorted iterators preserves order. Document this explicitly.

### 2. Add count helpers

## Acceptance Criteria

### Tests That Must Pass

1. **entities() sorted**: creating entities in arbitrary order, `entities()` yields them sorted by EntityId.
2. **entities() excludes archived**: archived entities do not appear in `entities()`.
3. **entities_of_kind filters correctly**: only entities of the requested kind are returned.
4. **entities_with_name**: returns only entities that have a Name component.
5. **query_name pairs**: returns `(EntityId, &Name)` tuples, sorted by EntityId.
6. **Multi-component intersection**: `entities_with_name_and_agent()` returns only entities with both components.
7. **query_name_and_agent tuples**: correct data in each tuple.
8. **Intersection excludes partial**: entity with Name but no AgentData is excluded from intersection.
9. **Filter preserves order**: filtering a query result maintains sorted order.
10. **Empty queries**: querying empty tables returns empty iterators without panic.
11. **entity_count correct**: matches number of live entities.
12. **Determinism**: two identical sequences of operations produce identical query results.
13. **Existing suite**: `cargo test -p worldwake-core` continues green.

### Invariants

1. Query iteration order is always deterministic (BTreeMap-sorted by EntityId).
2. Queries only return live entities unless explicitly requesting archived ones.
3. No query method provides mutable access to components.
4. Filter does not reorder results.
