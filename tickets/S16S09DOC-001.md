# S16S09DOC-001: Docs — Golden Reassessment And Causal-Boundary Guidance

**Status**: PENDING
**Priority**: MEDIUM
**Effort**: Small
**Engine Changes**: None
**Deps**: `archive/tickets/completed/S16S09GOLVAL-002.md`, `tickets/README.md`, `docs/golden-e2e-testing.md`, `AGENTS.md`

## Problem

The `S16S09GOLVAL-002` work exposed two process gaps:

1. a ticket can still be written as "tests only" even when that is only a hypothesis and current code may contradict the intended invariant, and
2. a golden can still be scoped around a broad downstream behavior when the cleaner architectural contract is an earlier causal boundary such as goal selection.

Those gaps encourage brittle tickets and over-broad goldens, which pulls the suite away from `docs/FOUNDATIONS.md` and toward scenario scripting.

## Assumption Reassessment (2026-03-20)

1. `tickets/README.md` already has strong reassessment, ordering, and scenario-isolation rules, but it does not explicitly say that `Engine Changes: None` or "tests only" is provisional until current code and focused coverage confirm there is no underlying architectural contradiction.
2. `docs/golden-e2e-testing.md` already prefers semantic assertion surfaces and warns against indirect proxies, but it does not explicitly tell authors to prefer the earliest causal boundary when that boundary is the actual contract. In practice, that leaves room for tickets to overreach into downstream execution when the invariant is really candidate/ranking/selection correctness.
3. `AGENTS.md` tells contributors to use decision traces first when debugging AI behavior, but it does not explicitly document the limitation discovered here: traces may show the selected outcome while still lacking full pressure provenance, so engineers may need focused lower-layer tests to localize the contradiction.
4. The current repo already has a good concrete example for the revised guidance: `archive/tickets/completed/S16S09GOLVAL-002.md` now documents why the correct assertion boundary is "first post-resolution selected goal is non-combat" rather than "agent eventually eats or heals."
5. This is a documentation/process ticket only. No production engine behavior should change.
6. This ticket is not ordering-centric, though it should clarify how to describe boundary choice when later effects are only downstream consequences of an earlier AI decision.
7. No heuristic/filter is being changed; this ticket only codifies better authoring and verification discipline.
8. Mismatch corrected: the existing docs are not wrong, but they are not explicit enough about provisional tests-only scoping, earliest-causal-boundary assertions, and the current limits of decision-trace provenance.

## Architecture Check

1. Tightening the documentation is the cleanest fix because the failure mode was partly authoring discipline, not just missing code. Good process should push future tickets toward the right abstraction boundary before implementation starts.
2. The guidance should remain principle-based and concrete, not a bag of one-off rules derived from this single scenario.
3. No backwards-compatibility shims, duplicate policy documents, or competing sources of truth should be introduced. The existing canonical files should be strengthened in place.

## Verification Layers

1. Ticket-authoring contract explicitly treats "tests only" as provisional -> `tickets/README.md`
2. Golden-testing guidance explicitly prefers earliest causal boundary over broader downstream consequences -> `docs/golden-e2e-testing.md`
3. AI debugging guidance explicitly states when decision traces are insufficient on their own and lower-layer focused tests are required -> `AGENTS.md`
4. Single-layer documentation ticket: no additional mixed-layer mapping is applicable beyond verifying the updated guidance is internally consistent and references real current examples

## What to Change

### 1. Tighten `tickets/README.md`

Add explicit guidance that:

1. `Engine Changes: None` is a provisional assumption until reassessment confirms no production contradiction,
2. tickets should be rewritten when new coverage exposes an underlying architectural bug,
3. if a golden can prove the intended contract at an earlier causal boundary, the ticket should not widen scope to later downstream behavior unless that later behavior is itself the contract.

### 2. Tighten `docs/golden-e2e-testing.md`

Add a short section or expand existing guidance to state:

1. prefer the earliest semantic boundary that proves the invariant,
2. do not couple a golden to later action execution when candidate generation, ranking, suppression, or selection is the real contract,
3. use `archive/tickets/completed/S16S09GOLVAL-002.md` as an example of narrowing from downstream behavior to the correct changed-conditions selection boundary.

### 3. Tighten `AGENTS.md`

Update the decision-trace debugging section to say explicitly:

1. decision traces are the first stop for AI reasoning,
2. if traces show the selected outcome but not the concrete world facts keeping that branch alive, drop to the shared lower-layer state/query tests instead of adding ad-hoc instrumentation,
3. a follow-up traceability ticket may be warranted when the missing information is architecturally important rather than merely inconvenient.
