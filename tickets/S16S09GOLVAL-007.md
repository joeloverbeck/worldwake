# S16S09GOLVAL-007: Golden Docs — Generate Inventory and Keep Coverage Docs in Sync

**Status**: PENDING
**Priority**: MEDIUM
**Effort**: Medium
**Engine Changes**: Yes — docs/tooling, no simulation behavior changes
**Deps**: `docs/golden-e2e-coverage.md`, `docs/golden-e2e-scenarios.md`, `docs/golden-e2e-testing.md`

## Problem

The golden E2E docs are currently hand-maintained, and they drift. Adding one new golden exposed that the suite counts, file summaries, and coverage descriptions were already partially stale.

That creates a documentation-traceability gap:

- the docs claim to be inventory-backed
- but there is no canonical generated inventory artifact or script
- updating the suite requires manually reconciling `rg`, `cargo test -- --list`, and prose summaries

For a suite that exists to lock down architectural promises, the inventory and coverage docs should derive from a cleaner source of truth.

## Assumption Reassessment (2026-03-21)

1. The current golden docs are hand-maintained. The relevant files are `docs/golden-e2e-coverage.md`, `docs/golden-e2e-scenarios.md`, and `docs/golden-e2e-testing.md`.
2. `docs/golden-e2e-coverage.md` explicitly claims its inventory was cross-checked against `cargo test -p worldwake-ai -- --list`, but the repository currently has no dedicated script or generated artifact that performs or records that reconciliation.
3. The recent spatial-planning addition required manual updates to file counts, suite totals, topology coverage wording, and scenario catalog entries. That is a repeatable source of drift and should be reduced by tooling rather than discipline alone.
4. This is a docs/tooling ticket, not a production simulation ticket. No authoritative-world, AI-planner, or action-handler behavior should change.
5. The right architectural boundary is generation assistance, not fully autogenerated prose. Counts, test-name inventory, and file-level summaries are good candidates for generation or script-backed validation; nuanced scenario explanations should remain authored text.
6. Ordering is not applicable here.
7. Not a stale-request, political, or `ControlSource` ticket.
8. No current script under `scripts/` appears to own the golden inventory or docs-sync problem. Reassessment found no existing dedicated tool for this workflow.
9. `docs/golden-e2e-testing.md` remains the right normative conventions document; this ticket should support it with better inventory hygiene, not replace it.
10. Scope correction: do not build a brittle parser that rewrites all narrative docs end-to-end. The clean solution is a narrow generated inventory/validation layer plus lightweight doc integration points.

## Architecture Check

1. The clean approach is to introduce a canonical inventory generator or validator for the mechanical facts:
   - `golden_*` counts
   - file-level test inventory
   - optionally per-file lists or suite totals
   Then keep narrative scenario prose authored by hand.
2. This respects Principle 25 from `docs/FOUNDATIONS.md` in spirit: derived summaries are caches, not truth. The generated inventory should be recomputable from the test files and `cargo test -- --list`, while the docs consume that derived summary.
3. No backwards-compatibility shims. Update the docs and workflow to the new path directly.

## Verification Layers

1. Inventory generation/validation is correct -> focused tooling test or deterministic script output checked against current `golden_*.rs` declarations
2. Coverage docs consume or reference the generated inventory cleanly -> documentation review plus command-based verification
3. Golden test suite names remain the authoritative source -> `cargo test -p worldwake-ai -- --list`
4. Single-layer note: this ticket does not change simulation behavior, so no action trace / decision trace / authoritative world-state mapping is applicable

## What to Change

### 1. Add a canonical golden inventory tool

Create a narrow script or tool under `scripts/` that produces the mechanical inventory facts for the golden suite, such as:

- total `golden_*` count
- per-file `golden_*` counts
- per-file test-name lists

It should read the current repository state and be deterministic.

### 2. Update docs to consume the inventory more cleanly

Revise the golden docs so the mechanically derived sections are either:

- generated into a checked-in artifact the docs reference, or
- validated by an explicit maintenance command documented in the files

The goal is that future suite changes require one obvious inventory refresh step instead of manual counting across multiple sources.

### 3. Document the maintenance workflow

Add a short maintainer workflow note explaining how to refresh or validate the golden inventory after adding/removing goldens.
